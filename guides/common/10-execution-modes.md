# 실행 모드 가이드

Good Vibe에는 **프로젝트 모드**와 **실행 모드** 두 가지 모드가 있습니다.
이름이 비슷해서 헷갈릴 수 있지만, 서로 다른 시점에 결정되는 별개 개념입니다.

---

## 프로젝트 모드 vs 실행 모드

|                | 프로젝트 모드 (`project.mode`)       | 실행 모드 (`executionState.mode`) |
| -------------- | ------------------------------------ | --------------------------------- |
| **언제 결정?** | `good-vibe:new`에서 복잡도 분석 시   | `good-vibe:execute` 시작 시       |
| **뭘 결정?**   | 팀 규모, 토론 라운드 수, 승인 방식   | Phase 진행 방식 (자동/수동)       |
| **선택지**     | quick-build, plan-execute, plan-only | interactive, semi-auto, auto      |
| **변경 가능?** | `good-vibe:new` 시 수동 선택 가능    | `good-vibe:execute` 시작 시 선택  |

---

## 프로젝트 모드 상세

프로젝트의 복잡도에 따라 워크플로우 전체 구조가 달라집니다.

### quick-build

간단한 프로젝트용. 토론 없이 바로 실행합니다.

- **팀 규모:** 2-3명 (CTO, Backend/Fullstack, QA)
- **토론:** 생략 (CTO 아키텍처 분석만)
- **승인:** 자동 (CEO 승인 불필요)
- **적합한 프로젝트:** 스크립트, 간단한 봇, 유틸리티

```
good-vibe:new → CTO 분석 → 실행 → QA 리뷰 → 완료
```

### plan-execute

중간 규모 프로젝트용. CTO+PO 빠른 분석 후 자동으로 실행까지 이어집니다.

- **팀 규모:** 3-5명
- **토론:** 생략 (CTO+PO 빠른 분석)
- **승인:** 자동 (분석 확인 후 바로 실행)
- **적합한 프로젝트:** 웹앱, API 서버, 중소형 서비스

```
good-vibe:new → CTO+PO 분석 → 자동 승인 → good-vibe:execute → 완료
```

### plan-only

대규모 프로젝트용. 충분한 토론 후 CEO가 직접 승인합니다.

- **팀 규모:** 5-8명
- **토론:** 최대 3라운드 (80% 수렴까지)
- **승인:** CEO 수동 승인 필요 (`good-vibe:approve`)
- **적합한 프로젝트:** 마이크로서비스, SaaS 플랫폼, 복잡한 시스템

```
good-vibe:new → good-vibe:discuss (수렴까지) → good-vibe:approve → good-vibe:execute → 완료
```

### 어떤 모드를 써야 할까?

| 상황                                         | 추천 모드                               |
| -------------------------------------------- | --------------------------------------- |
| "빨리 만들어봐야지"                          | quick-build                             |
| "제대로 만들고 싶지만 기획에 시간 쏟긴 싫어" | plan-execute                            |
| "아키텍처부터 꼼꼼하게 검토하고 싶어"        | plan-only                               |
| "잘 모르겠어"                                | `good-vibe:new`가 복잡도 보고 자동 추천 |

## 실행 모드 상세

`good-vibe:execute` 시작 시 Phase 진행 방식을 선택합니다.

### interactive

Phase마다 CEO에게 진행 여부를 확인합니다.

- 각 Phase 완료 후 "다음 Phase 진행?" 질문
- Phase별 지침(phaseGuidance)을 직접 전달 가능
- 리뷰 후 개입 가능 (`reviewIntervention` 설정 시)

**추천:** 처음 Good Vibe를 사용할 때. 각 Phase가 어떻게 진행되는지 확인하면서 진행할 수 있습니다.

### semi-auto

배치 단위(기본 3 Phase)마다 확인합니다.

- 3개 Phase가 끝나면 CEO에게 확인
- 에스컬레이션(수정 2회 실패) 시에도 멈춤

**추천:** Phase가 많은 프로젝트에서 매번 확인하기 번거로울 때.

### auto

문제가 생길 때만 멈춥니다.

- 에스컬레이션(수정 2회 실패) 시에만 CEO에게 질문
- 나머지는 전부 자동 진행

**추천:** Good Vibe에 익숙해진 후, 또는 SDK에서 프로그래밍으로 사용할 때.

### 어떤 모드를 써야 할까?

| 상황                                      | 추천 모드   |
| ----------------------------------------- | ----------- |
| Good Vibe 처음 사용                       | interactive |
| 과정을 지켜보고 싶지만 매번 확인은 귀찮아 | semi-auto   |
| 결과만 중요해, 과정은 알아서              | auto        |
| SDK로 자동화 파이프라인 구축              | auto        |

## 조합 예시

프로젝트 모드와 실행 모드는 자유롭게 조합할 수 있습니다.

| 프로젝트 모드              | 실행 모드                                 | 시나리오 |
| -------------------------- | ----------------------------------------- | -------- |
| quick-build + auto         | 가장 빠른 경로. 간단한 봇을 완전 자동으로 |
| quick-build + interactive  | 간단하지만 과정을 확인하고 싶을 때        |
| plan-execute + semi-auto   | 중간 규모 프로젝트의 일반적인 선택        |
| plan-execute + interactive | 중간 규모지만 꼼꼼하게 확인하고 싶을 때   |
| plan-only + interactive    | 대규모 프로젝트를 Phase별로 검토          |
| plan-only + auto           | 기획은 꼼꼼하게, 실행은 자동으로          |

## 에스컬레이션

어떤 실행 모드에서든, 수정이 2회 실패하면 CEO에게 판단을 요청합니다.

| 선택지              | 의미              | 언제 선택?                          |
| ------------------- | ----------------- | ----------------------------------- |
| **계속** (continue) | 추가 수정 시도    | 핵심 기능이라 반드시 성공해야 할 때 |
| **건너뛰기** (skip) | 이 Phase를 건너뜀 | 부가 기능이라 나중에 추가해도 될 때 |
| **중단** (abort)    | 실행 전체 중단    | 기획 자체를 재검토해야 할 때        |

"계속"을 선택하면 수정 방향을 직접 지시할 수 있습니다 (ceoGuidance).

## 실행 옵션

`good-vibe:execute` 시작 시 실행 모드 외에 추가 옵션을 선택할 수 있습니다.

### 팀 커뮤니케이션

plan-execute / plan-only 프로젝트에서만 표시됩니다 (quick-build는 팀이 작아 생략).

| 설정                   | 동작                                                     |
| ---------------------- | -------------------------------------------------------- |
| **기본 실행** (기본값) | 태스크 실행 → 리뷰 → 품질 게이트. 빠르고 비용 적음       |
| **팀 커뮤니케이션 ON** | 리뷰 대화 + 전문가 협의 활성화. 품질 향상, LLM 호출 추가 |

**팀 커뮤니케이션 ON일 때 추가되는 기능:**

- **리뷰 대화**: 리뷰어가 궁금한 점을 구현자에게 질문 → 구현자가 답변 → 답변을 반영한 최종 리뷰. 단순 지적이 아닌 맥락을 이해한 리뷰가 됩니다.
- **전문가 협의**: 태스크 실행 중 에이전트가 다른 역할의 전문가에게 질문할 수 있습니다. 예: Backend 개발자가 Security 전문가에게 인증 방식을 질문.

**추천:**

- 빠른 결과가 중요하면 → 기본 실행
- 품질이 중요한 프로젝트 → 팀 커뮤니케이션 ON

### 멀티 AI 리뷰

구현자와 다른 AI 모델로 리뷰하여 다양한 관점을 확보합니다.

| 항목 | 설명                                                      |
| ---- | --------------------------------------------------------- |
| 기본 | Claude만으로 구현 + 리뷰                                  |
| 멀티 | 구현은 Claude, 리뷰에 Gemini 등 다른 모델을 추가 배정     |
| 조건 | Gemini CLI 설치 + 인증 필요 (`gemini --version`으로 확인) |

`good-vibe:hello`에서 Gemini CLI가 감지되면 자동으로 제안됩니다.
Gemini가 없으면 이 옵션은 표시되지 않으며, Claude만으로 정상 동작합니다.
